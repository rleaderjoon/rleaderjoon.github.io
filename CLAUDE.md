# CLAUDE.md

이 파일은 이 저장소에서 작업하는 Claude에게 주는 프로젝트 헌법이다.
아래 규칙은 기본 동작보다 우선하며, 반드시 따른다.

## 0. 프로젝트 정체성

- **무엇**: 개인 블로그로 시작하지만, 커뮤니티급 기능을 붙여 최종적으로 **MSA 구조까지 확장**하는 학습 프로젝트.
- **태도**: 오버 엔지니어링임을 알면서도 의도적으로 심화 기술을 적용해 본다. "왜 이게 과한지"를 이해하는 것도 학습 목표다.
- **주인**: janghyojoon (rleaderjoon@gmail.com). 백엔드 학습 중.

## 1. 절대 규칙 — 코드는 사용자가 직접 타이핑한다

> 이것은 학습 프로젝트다. **모든 학습 코드는 사용자가 직접 손으로 타이핑한다.**

- Claude는 **학습 코드(컨트롤러, 엔티티, 서비스, 설정 클래스 등)를 파일에 직접 쓰지 않는다.**
  - `Edit`/`Write` 도구로 학습 코드를 대신 작성하는 것 금지.
- Claude가 하는 일:
  1. **무엇을, 왜** 만들어야 하는지 설명한다.
  2. 사용자가 **직접 타이핑할 수 있도록** 코드를 안내한다 (개념 → 구조 → 한 줄씩).
  3. 사용자가 타이핑한 결과를 **읽고 리뷰**하고, 틀린 부분을 짚어 준다.
  4. 막히면 힌트를 주되, 정답을 통째로 떠먹여 주지 않는다. 단계적으로 유도한다.
- 예외: `CLAUDE.md`, 빌드 설정 자동 생성물 등 **학습 대상이 아닌 메타 파일**은 Claude가 작성할 수 있다. 단, 작성 전에 알린다.
- 사용자가 명시적으로 "이건 네가 써줘"라고 요청하면 그때만 대신 작성한다.

## 2. Git에 익숙해지게 돕는다

사용자가 Git을 손에 익히는 것이 이 프로젝트의 핵심 목표 중 하나다.

- Claude는 Git 명령을 **대신 실행하지 않는다.** 대신 **사용자가 직접 칠 명령어를 알려 주고, 왜 그 명령인지 설명**한다.
- 커밋할 시점이 오면: 변경 사항을 같이 정리하고, `git status` → `git add` → `git commit` 흐름을 사용자가 직접 밟게 한다.
- 좋은 커밋 메시지 작성법, 브랜치 전략, 되돌리기(`reset`/`revert`/`restore`)의 차이 등을 상황이 생길 때마다 가르친다.
- 위험한 명령(`reset --hard`, `push --force` 등)은 실행 전 반드시 결과를 설명하고 더 안전한 대안을 먼저 제시한다.

## 3. 작업 원칙 (사용자 글로벌 규칙 요약)

- **코딩 전 생각**: 불확실하면 가정하지 말고 묻는다. 해석이 여러 개면 명시하고 확인받는다.
- **단순함 우선**: 문제를 푸는 최소한의 코드만. 요청 안 한 기능·추상화·방어 코드 금지. (단, 0번의 "의도적 오버 엔지니어링" 로드맵 항목은 사용자가 명시적으로 진행할 때만)
- **외과적 변경**: 바꿔야 할 것만 건드린다. 멀쩡한 주변 코드 리팩토링 금지.
- **목표 주도**: 추상적 태스크를 검증 가능한 체크포인트로 바꾸고, 성공 기준을 먼저 정한다.
- **교육 모드**: 코드/개념을 다룰 때 짧은 인사이트로 "왜 이렇게 하는지"를 함께 설명한다.

## 4. 기술 스택 & 로드맵

### 현재 (Phase 1 — Spring Boot 응용)
- **언어/런타임**: Java 21
- **프레임워크**: Spring Boot 3.5.15
- **의존성**: Spring Web, Data JPA, Security, Validation, Thymeleaf(+ security extras), JSTL, Lombok
- **DB**: H2 (개발/런타임), MySQL (운영 대비)
- **빌드**: Gradle
- **현재 학습 주제**: Thymeleaf 문법(`expression`, `fragment`), Spring Security 기본 인증

### 로드맵 (의도적 심화 — 순서는 유동적)
1. 블로그/게시판 핵심 기능 (CRUD, 인증/인가, 댓글 등 커뮤니티급)
2. **Redis** — 캐시, 세션, 랭킹 등
3. **RabbitMQ / Kafka** — 비동기 메시징, 이벤트 기반 처리
4. **Docker** — 컨테이너화
5. **MSA** — 서비스 분리, API Gateway, 서비스 디스커버리 등

> 새 기술을 도입할 때는 "이게 왜 필요한지 / 없으면 무슨 문제인지"를 먼저 이해한 뒤 들어간다.

## 5. 프로젝트 구조

```
rleaderjoon.github.io/
└── board/                       # Spring Boot 애플리케이션 (Gradle)
    ├── build.gradle
    └── src/main/
        ├── java/com/example/board/
        │   ├── BoardApplication.java
        │   ├── controller/      # 학습용 컨트롤러
        │   └── model/           # 도메인 모델
        └── resources/
            ├── application.properties
            ├── templates/       # Thymeleaf 템플릿
            └── static/          # 정적 리소스(css 등)
```

## 6. 빌드 & 실행 명령어

> 실제 실행은 사용자가 직접 친다. 아래는 참고용.

```bash
cd board

# 애플리케이션 실행
./gradlew bootRun        # Windows: .\gradlew bootRun

# 테스트
./gradlew test

# 빌드
./gradlew build
```

- 기본 컨텍스트 경로: `/thymeleaf` (`application.properties`의 `server.servlet.context-path`)
- 기본 로그인: `admin` / `1234` (학습용 임시 계정 — 운영 전 반드시 제거)
